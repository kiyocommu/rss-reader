RSS Reader 取扱説明書

RSS Reader は、GitHub Pages などの静的ホスティングで公開して使う、ブラウザベースのRSSリーダーです。 フィード一覧、ピン、Read It Later をブラウザに保存し、Supabaseを設定した場合は複数端末で共有できます。

このHTML版は、人間が読みやすい章立てと、AIが仕様を抽出しやすい構造化セクションを両方含みます。

単体で動作`index.html` を開くだけでローカルRSSリーダーとして使えます。
同期は任意Supabase URL、anon key、共有IDを設定した時だけクラウド同期します。
個人利用向け共有IDを知る人は同じデータを読み書きできます。完全な認証ではありません。

最短セットアップ

ローカルだけで使う

  1. output/rss-reader/index.html をブラウザで開く。
  2. URL入力欄にRSSフィードURLまたはサイトURLを入れる。
  3. add を押す。
  4. ヘッダーの RSS Reader ⟳ または r で記事を取得する。
  5. 必要なら ブックマークレット設定.html を開き、ワンクリック追加用のブックマークレットを登録する。

Supabase未設定時は、フィード、ピン、保存記事がそのブラウザの localStorage にだけ保存されます。

複数端末で同期する

  1. SupabaseでProjectを作成する。
  2. supabase-schema.sql を Supabase SQL Editor で実行する。
  3. output/rss-reader/ を GitHub Pages などで公開する。
  4. RSS Reader の設定画面で Supabase URL、anon key、共有IDを入力する。
  5. 保存 または 同期 を押す。

同期したい端末では、同じSupabase設定と同じ共有IDを入力します。

重要: 共有IDはパスワードではありません。共有IDとanon keyを知っている人は、その共有IDのデータを読み書きできます。 個人利用・限定共有向けの設計です。

フィード操作

フィードを追加する

1
URL入力欄に、RSSフィードURLまたはサイトURLを入力します。
2
add を押します。入力欄で Enter を押しても追加できます。
3
複数フィードが見つかった場合は、追加したいフィードにチェックを入れて 追加 を押します。
4
記事を取得したい場合は、ロゴの RSS Reader ⟳ または r で更新します。

クリップボードにURLがある場合は、入力欄左の 📋 ボタンで貼り付けできます。ブラウザの権限が必要な場合があります。

フィードを更新する

操作動作
ヘッダーのロゴをクリック登録済みの全フィードを更新します。
r全フィードを更新します。
一覧を先頭で下へ引っ張る「すべてのフィード」表示中だけ、Pull-to-Refreshで全フィードを更新します。
個別フィードの ⟳選択中のフィードだけ更新します。

フィードを削除する

  1. 記事一覧上部のフィード選択で削除したいフィードを選びます。
  2. 表示されたフィード操作バーの × を押します。
  3. 削除すると、そのフィード由来の記事キャッシュも一覧から外れます。

ブックマークレット

旧RSS Readerにあったブックマークレット設定は、RSS Reader向けに ブックマークレット設定.html として同梱しています。 閲覧中のページからRSSフィードを検出し、RSS Readerの index.html?add=... を開いて自動追加します。

  1. output/rss-reader/ブックマークレット設定.html をブラウザで開きます。
  2. 📡 RSS Readerに追加 をブックマークバーへドラッグします。
  3. 追加したいサイトを開き、登録したブックマークレットをクリックします。
  4. フィードが見つかった場合はRSS Readerが新しいタブで開き、自動追加されます。

ページ内にRSSリンクがない場合は、確認後にページURLをRSS Readerへ送れます。 RSS Reader側でサイトURLからフィード候補を探索します。

記事を読む

  • 記事一覧で記事をクリックすると、記事詳細に本文が表示されます。
  • 記事タイトルをクリック、または o を押すと元記事を新しいタブで開きます。
  • 24時間以内の新着記事は日時が強調表示されます。
  • 更新で新しく取得された記事には NEW バッジが付きます。ページ再読み込み後は消えます。
  • 記事の最後にある終端表示をクリックすると「すべて読みました」の画面になります。

ピン / Read It Later

ピン

気になる記事URLに一時的な印を付ける機能です。記事一覧、記事詳細、ショートカット p から切り替えられます。

  • PCでは上部のピンバーからピン記事を確認できます。
  • スマホでは右上の 📍 ボタンからモーダルで確認します。
  • ピン記事モーダル内で、ピン解除やRead It Later追加もできます。

Read It Later

後で読みたい記事を本文付きで保存する機能です。記事詳細の保存ボタン、記事一覧の保存ボタン、または s で切り替えます。

  • フィード選択の Read It Later で保存記事だけを表示できます。
  • b でRead It Later表示と元の表示を切り替えます。
  • 一覧上部の export で Tab Bookmark Manager 向けJSONを書き出せます。

Supabase同期

RSS Reader は、Supabaseをフィード、ピン、Read It Later の共有保存先として使います。 記事の取得済みキャッシュ、テーマ、表示モード、パネル幅、プロキシ成功履歴は端末ごとのローカル設定です。

同期されるもの

対象Supabaseテーブル主なキー
フィード一覧rss_feedsprofile_id + url
ピンした記事URLrss_pinsprofile_id + link
Read It Laterの記事rss_saved_articlesprofile_id + link

Supabaseから取得するタイミング

  • 起動時にSupabase設定があり、かつそのブラウザのローカルにフィードがない場合。
  • 設定画面で 保存 を押し、ローカルにフィードがない場合。
  • 設定画面で 同期 を押した場合。ローカルにフィードがあれば先に保存し、その後に読み戻します。
ローカルにすでにフィードがある端末では、起動時の自動取得を行いません。空または古いクラウドデータで、手元のフィードを不用意に上書きしないためです。

Supabaseへ保存するタイミング

  • フィードを追加・削除した時。
  • ピンを追加・解除した時。
  • Read It Laterに保存・解除した時。
  • importでフィード、ピン、Read It Laterを読み込んだ時。
  • 設定画面で 同期 を押し、ローカルにフィードがある時。

通常操作では、変更直後に短い待ち時間を置いて自動保存します。短時間に複数操作した場合は、最後の状態をまとめて保存します。

保存方式

  1. 現在のローカル状態にある行をSupabaseへ追加または更新します。
  2. Supabase上にあるが、現在のローカル状態にない行だけを削除します。
  3. 保存後の件数を読み戻し、期待件数と一致するか確認します。

この方式により、保存途中の通信失敗でクラウド側が空になりにくくなっています。ただし、複数端末の同時変更を細かくマージする仕組みではありません。

複数端末で安全に使う

3台以上で使う場合は、最後にSupabaseへ保存した端末のローカル状態がクラウド側へ反映されます。 古い状態の端末で操作すると、その端末にないフィード、ピン、Read It LaterがSupabase上で削除対象になることがあります。

安全な運用

  • 新しい端末で使い始める時は、ローカルにフィードがない状態でSupabaseから取得します。
  • 久しぶりに使う端末では、古いローカル状態のまま編集しないでください。
  • 複数端末で同時にフィード追加、ピン、Read It Later操作をしないでください。
  • 大きく整理する前は、ヘッダーの export でバックアップを取ってください。
  • どの端末が最新か分からない場合は、最新だと分かっている端末で先に 同期 してから他端末を使います。

上書きリスクの例

  1. PCでRead It Laterに10件保存し、同期済み。
  2. iPhone側は古い状態でRead It Laterが3件だけ残っている。
  3. iPhone側で保存解除や 同期 を行う。
  4. iPhone側の3件が現在の状態として保存され、PCで追加した残り7件は削除対象になる可能性があります。

スマホでの使い方

スマホ幅では1カラム表示になります。記事一覧と記事詳細は画面を切り替える形で表示されます。

操作動作
URL ボタンRSSフィードURL入力欄を開閉します。
Feed ボタンフィード選択を開閉します。
📍 ボタンピン記事モーダルを開きます。ピンがある時は件数が表示されます。
記事をタップ記事詳細へ切り替わります。
記事詳細の左端から右へスワイプ記事一覧へ戻ります。
左下の 前 / 次前後の記事へ移動します。
前 / 次ボタン左側の持ち手をドラッグナビゲーション位置を移動します。位置は端末ごとに保存されます。
一覧先頭から下へ引っ張る「すべてのフィード」表示中だけ全フィード更新します。

スマホの記事詳細では、ロゴ、設定、URL入力欄を隠して本文領域を広くします。記事タイトル自体が元記事へのリンクです。

import / export

バックアップJSON

ヘッダーの export は、フィード、ピン、Read It LaterをまとめてJSON保存します。ファイル名は rss-reader-backup-YYYY-MM-DD.json です。

{
  "feeds": [
    {"url": "https://example.com/feed.xml", "title": "Example", "color": "#e94560"}
  ],
  "pinned": ["https://example.com/article"],
  "saved": [
    {
      "title": "記事タイトル",
      "link": "https://example.com/article",
      "feedUrl": "https://example.com/feed.xml",
      "pubDate": "2026-05-28T00:00:00.000Z",
      "image": "",
      "content": "本文HTML",
      "savedAt": "2026-05-28T00:00:00.000Z"
    }
  ]
}

ヘッダーの import はこの形式を読み込みます。読み込み後は記事キャッシュを空にするため、更新ボタンで記事を取得してください。

Read It Laterのエクスポート

記事一覧上部の export は、Read It Laterの記事を Tab Bookmark Manager 向けJSONとして書き出します。ファイル名は rss-read-later-YYYY-MM-DD.json です。

トラブル対応

permission denied for table rss_feeds が出る

Supabase SQL Editorで最新版の supabase-schema.sql をもう一度実行してください。テーブル作成済みでも、anon ロールへの grant select, insert, update, delete やRLS policyが不足している場合があります。

TypeError: Load failed がiPhone Chromeだけで出る

PCで同期できている場合、Supabase権限ではなくiOSブラウザ側のネットワーク取得失敗の可能性があります。VPN、広告ブロック、プライベートリレー、コンテンツブロッカーがSupabase RESTエンドポイントを止めていないか確認してください。

同期しました（フィード0件 ...） と表示される

そのブラウザのローカルデータにフィードがありません。GitHub Pages、localhost、file:// は別々の保存領域です。フィードがある端末で先に同期してください。

同期データを取得しましたが、フィードは0件です と表示される

Supabase上のその共有IDにフィードが保存されていません。共有IDが合っているか、または同期元の端末でフィード0件として保存していないか確認してください。

起動時にクラウド内容が反映されない

ローカルにフィードがある端末では、空データで上書きしないため起動時取得を止めます。設定画面の 同期 を押してください。

フィード取得が失敗する

公開CORSプロキシのレート制限、サイト側のブロック、XML形式の問題が考えられます。必要に応じて設定画面でカスタムCORSプロキシを指定してください。EUC-JPなど非UTF-8フィードも、カスタムプロキシ経由で正しく処理されるようにしています。
